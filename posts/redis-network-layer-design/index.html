Redis网络层设计解析

July 18, 2026 · 3 min

解析 Redis 网络层的设计思路，辩证地看待每个选择的优劣。

1. 为什么 Redis 选择 TCP？#

TCP 的适用场景：长连接、大数据量、数据可靠性要求高。Redis 完全符合这三个场景。

场景Redis 的匹配
长连接客户端与服务端保持持久连接，避免频繁握手
大数据量批量操作、Pipeline、主从同步都可能传输大量数据
可靠性数据不能丢，写命令必须可靠到达

TCP 的缺点：数据实时性差、短连接不友好——这也是 HTTP/3 舍弃 TCP 改用 QUIC 的原因。

2. 为什么说应用层视角下 TCP 是"异步"的？#

这是一个反常识的点。

TCP 在传输层是同步的——确认帧、重传机制、滑动窗口、拥塞控制，教材里明摆着写着 TCP 是同步可靠的。

那为什么应用层视角下它是"异步"的？

答案在于系统调用语义。

应用层
  │  send()  ──→  数据拷进内核发送缓冲区  ──→  返回（不管发没发出去）
  │  recv()  ──→  从内核接收缓冲区取数据  ──→  返回（不管到没到齐）
  │
  ▼
内核层
  └── TCP 栈：什么时候发、分几个包、重不重传——全是内核说了算
  • send() 只承诺"数据拷进内核发送缓冲区了"，不承诺"数据发出去了"。什么时候真正发、分几个包发、重不重传，全是内核 TCP 栈说了算——应用层完全不知情。这就是异步：你把活交给内核，它什么时候干完不告诉你。
  • recv() 同理：数据什么时候到、分几次到、一次 recv 能拿多少，全不归你管。

粘包与拆包#

更麻烦的是，TCP 是字节流，不保留消息边界：

发送端： write("hello")  × 3

接收端可能拿到：
  ┌─────────────────────┐
  │ "hellohellohello"    │  ← 粘包（三次合并成一次）
  │ "hel" + "lohello..." │  ← 拆包（一次拆成多次）
  └─────────────────────┘

你需要一个应用协议来界定消息边界。

3. 应用层消息界定格式#

方式原理优点缺点
分隔符界定特殊字节标记消息结尾（如 \r\n）简单，可读性好不适用于二进制（数据中可能出现分隔符）
前置长度固定字节 header 声明消息体长度简单，适合二进制需要提前知道长度
固定长度所有消息字节数完全相同实现最简单浪费空间，灵活性差

4. RESP 格式#

Redis 选择了混合策略：首字符界定类型 + 分隔符 + 前置长度按需组合。

首字节类型格式示例
+Simple String+<文本>\r\n+OK\r\n
-Error-<错误信息>\r\n-ERR unknown command\r\n
:Integer:<数字>\r\n:1000\r\n
$Bulk String$<长度>\r\n<数据>\r\n$5\r\nhello\r\n
*Array*<元素数>\r\n<元素>...*2\r\n$3\r\nfoo\r\n$3\r\nbar\r\n

设计思路：

  • 首字符用来快速区分类型，一个字节完成分派
  • 短文本场景（Simple String、Error、Integer）：\r\n 分隔符足够简洁，一次界定边界的消耗几乎忽略不计，也不会有二进制问题
  • 二进制/长串场景（Bulk String）：使用前置长度，简单可靠，适合任意二进制数据

这就是 RESP 的精妙之处——不是死板地选一种，而是按类型选用最合适的策略。

while (1) {
    FD_ZERO(&rfds);
    FD_SET(fd1, &rfds);   // 每次循环都要重新注册所有 fd
    FD_SET(fd2, &rfds);
    // ...

    select(maxfd + 1, &rfds, NULL, NULL, NULL);

    for (int i = 0; i <= maxfd; i++) {   // 遍历所有 fd
        if (FD_ISSET(i, &rfds)) {

        }

两个致命问题：

  1. 每次都要重新传递 fd 集合 — 内核与应用层之间大量数据拷贝
  2. O(n) 遍历 — select 返回后必须遍历所有 fd 才能找到就绪的那几个，大部分遍历是无效的

epoll 如何解决：事件驱动#

epoll 不会让你去主动遍历寻找，而是直接告诉你哪些 fd 触发了事件。

epoll fd 内部结构：

  ┌── 红黑树 (rbtree) ───── 存储所有注册的 fd，key = fd
  │                          （用于 epoll_ctl 的增删改查）
  │
  └── 就绪链表 (rdllist) ─── 存储就绪的 fd
                              （epoll_wait 直接从这里取，O(1)）

工作流程：

网卡中断 → 内核将数据放入缓冲区
         → 调用 ep_poll_callback() 回调
         → 将 fd 丢进 rdllist（就绪链表）
         → epoll_wait 返回，直接取 rdllist 中的 fd
selectpollepoll
数据结构位图 (fd_set)链表红黑树 + 就绪链表
fd 上限1024（可改但重编译）无限制无限制
每次注册 fd是（重新传整个集合）是否（增量增删，epoll_ctl）
获取就绪 fdO(n) 遍历O(n) 遍历O(1) 直接从就绪链表取
内核-用户态拷贝大量大量只拷贝就绪的
设计思想轮询轮询事件驱动（观察者模式）

epoll 的设计思想和设计模式中的观察者模式本质相同——“别问我，我通知你”。

6. 水平触发 vs 边沿触发：Redis 为什么选择水平触发（LT）？#

水平触发 (LT)边沿触发 (ET)
触发条件缓冲区有数据就通知缓冲区状态变化时才通知一次
半包处理数据没读完下次继续通知数据没读完不会再通知，直到新数据到达
实现复杂度简单需要循环读直到 EAGAIN

快速理解水平触发和边沿触发：

Redis 选 LT 的原因#

边沿触发遇到半包会导致数据饥饿：如果一次没读完，fd 不再被触发，剩余数据就"卡"在里面了——直到新数据到来才会被重新触发。

对于 Redis 这种单线程模型来说，任何阻塞或饥饿都是致命的。Redis 在其他地方的设计也能看出它在主动规避阻塞：

场景规避策略
网络 I/O事件驱动 + 水平触发
持久化fork() + COW 写时复制（子进程干活，主进程不阻塞）
数据读写单线程，避免锁争夺开销
日志写入重定向符异步写入
惰性删除随机抽样，在主循环中顺手做，不交给中断

水平触发的好处：实现简单 + 避免半包阻塞。在 Redis 单线程模型下，LT 的额外唤醒开销远小于 ET 饥饿带来的风险。

7. 单线程 Reactor 模型#

把前面讲的串起来——epoll（事件分离） + 非阻塞 IO + 单线程执行，就是标准的 Reactor 模式：

init_server()
register_events(listen_fd)

while (1)
    n = epoll_wait(events)         // 同时等所有 fd（accept + 已连接 socket）
    for (0..n)
        if (is_listen_fd)          // 新连接
            cfd = accept()
            epoll_ctl(ADD, cfd)    // 注册进红黑树
        else                       // 已有连接
            handle_read()          // 读 → 解析 RESP → 执行 → 写回

Reactor 三个角色：

角色对应实现
事件分离器 (Demultiplexer)epoll — 同时监听所有 fd，谁就绪通知谁
事件分发器 (Dispatcher)事件循环 while(1) { epoll_wait → for each }
事件处理器 (Handler)handle_read() — 解析 RESP、路由命令、执行、写回

为什么单线程就够了？

Redis 的瓶颈不在 CPU（纯内存操作，微秒级），而在网络和内存带宽。单线程避免了锁开销、上下文切换，一条命令从头到尾不被打断——代码简单，bug 少。

这里的"单线程"指命令执行线程。Redis 6.0+ 引入了 IO 线程（只处理网络读写），但命令执行仍然是单线程的。

8. 总结#

从 TCP 字节流怎么切出消息，到 epoll 怎么同时盯上万个连接，整个设计链条是逐层递进的：

TCP 字节流（粘包/拆包）
    │  问题：消息边界在哪？
    ▼
RESP 格式（分隔符 + 前置长度，二进制安全）
    │  问题：怎么高效同时等上万个连接？
    ▼
IO 多路复用 — epoll（O(1) 取就绪事件，观察者模式）
    │  问题：怎么把这些组织成一个高效的调度模型？
    ▼
单线程 Reactor（事件分离 + 分发 + 处理，一帧到底）

每一层解决上一层的问题，缺一环都撑不起 Redis 的单机十万 QPS。
