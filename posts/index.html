Redis过期删除机制：从惰性删除到定期抽样

解析 Redis 过期 key 删除策略的层层设计——从惰性删除到定期抽样，理解"宁可少删，不可阻塞"的哲学。 1. 过期删除的三种设计思路 设计 key 过期机制，通常有三种方案： 方案一：定时全量扫描 维护一张 TTL 表，定时扫描表中哪些 key 过期了。 问题：O(n) 遍历，极其低效。百万级 key 扫一轮，CPU 全耗在上面了。 方案二：系统中断驱动 利用系统调用的定时器中断，精确在 key 过期的那一刻触发删除。 问题：看似优雅，但对 Redis 这种海量 key 的内存数据库来说，未知的中断 = 不可预知的阻塞。频繁中断也会严重影响系统吞吐，违背 Redis 单线程高性能的设计哲学。 方案三：惰性删除（Redis 的选择） 不主动清理过期 key，只在访问时顺手看一眼有没有过期。 优点：几乎没有额外 CPU 开销。 缺点：用空间换时间——已经过期但长期未被访问的 key 会一直占着内存，成为"内存垃圾"。 2. 惰性删除的兜底：定期抽样删除 惰性删除本身的缺陷是内存回收不及时，所以需要一个兜底机制来主动扫描。但这又回到了方案一的问题——全量扫描太重。 Redis 的解法是定期 + 抽样：不扫全部，每次只随机抽一小部分，用概率换取性能。 3. Redis网络层设计解析

解析 Redis 网络层的设计思路，辩证地看待每个选择的优劣。 1. 为什么 Redis 选择 TCP？ TCP 的适用场景：长连接、大数据量、数据可靠性要求高。Redis 完全符合这三个场景。 场景 Redis 的匹配 长连接 客户端与服务端保持持久连接，避免频繁握手 大数据量 批量操作、Pipeline、主从同步都可能传输大量数据 可靠性 数据不能丢，写命令必须可靠到达 TCP 的缺点：数据实时性差、短连接不友好——这也是 HTTP/3 舍弃 TCP 改用 QUIC 的原因。 2. 为什么说应用层视角下 TCP 是"异步"的？ 这是一个反常识的点。 TCP 在传输层是同步的——确认帧、重传机制、滑动窗口、拥塞控制，教材里明摆着写着 TCP 是同步可靠的。 那为什么应用层视角下它是"异步"的？ 答案在于系统调用语义。 应用层 │ send() ──→ 数据拷进内核发送缓冲区 ──→ 返回（不管发没发出去） │ recv() ──→ 从内核接收缓冲区取数据 ──→ 返回（不管到没到齐） │ ▼ 内核层 └── TCP 栈：什么时候发、分几个包、重不重传——全是内核说了算 send() 只承诺"数据拷进内核发送缓冲区了"，不承诺"数据发出去了"。什么时候真正发、分几个包发、重不重传，全是内核 TCP 栈说了算——应用层完全不知情。这就是异步：你把活交给内核，它什么时候干完不告诉你。 recv() 同理：数据什么时候到、分几次到、一次 recv 能拿多少，全不归你管。 粘包与拆包 更麻烦的是，TCP 是字节流，不保留消息边界： ...

从竞赛到系统编程：我为什么在三个国奖后转型

三个国奖之后 蓝桥杯国一、天梯赛个人国二、睿抗国一——打到第三个的时候，我开始意识到一个问题：再做一道 DP、再刷一个数据结构，能带来的提升越来越小了。 竞赛的边际收益在递减。这不是"刷够 500 题就能质变"的阶段了——那个阶段早在一年前就过去了。现在是技术天花板的问题：我不太可能靠继续刷题冲进 ICPC 区域赛金牌。 那与其在同一个维度上内卷，不如切换到另一个维度。 为什么是底层 竞赛训练了我的算法直觉——复杂的区间合并、懒标记下传、状态机 DP——这些都能在几十分钟内拆解清楚。但有个问题一直让我不舒服： 我不知道 std::vector 怎么分配内存的。 不知道 sort 的快排和堆排怎么切换的。不知道程序从 main 到 cout 中间经历了什么。 我写的所有代码，都运行在一个我从未认真审视过的"黑盒"之上。 这不只是好奇心——它是能力边界。不理解底层，我就永远写不出高性能的系统，没法 debug 一个内存泄漏到崩溃的程序，也看不懂那些真正决定软件质量的代码。 路线 按 CS 经典路线走，项目驱动 + 博客记录： CS:APP → OSTEP → 选方向深耕（OS / 网络 / 编译） 关于这个博客 接下来的文章会是边做边记的学习笔记——读一章原理，写一段代码，记录遇到的问题和思考。 不是写给谁看的教程，只是"我理解了什么"的记录。但如果你也在从竞赛转向系统编程，希望这些笔记能帮到你。
