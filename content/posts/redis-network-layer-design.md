## 1. 为什么 Redis 选择 TCP？

TCP 的适用场景：**长连接、大数据量、数据可靠性要求高**。Redis 完全符合这三个场景。

| 场景 | Redis 的匹配 |
|------|-------------|
| 长连接 | 客户端与服务端保持持久连接，避免频繁握手 |
| 大数据量 | 批量操作、Pipeline、主从同步都可能传输大量数据 |
| 可靠性 | 数据不能丢，写命令必须可靠到达 |

TCP 的缺点：数据实时性差、短连接不友好——这也是 HTTP/3 舍弃 TCP 改用 QUIC 的原因。

## 2. 为什么说应用层视角下 TCP 是"异步"的？

这是一个**反常识**的点。

TCP 在**传输层**是同步的——确认帧、重传机制、滑动窗口、拥塞控制，教材里明摆着写着 TCP 是同步可靠的。

那为什么**应用层视角**下它是"异步"的？

**答案在于系统调用语义。**

```
应用层
  │  send()  ──→  数据拷进内核发送缓冲区  ──→  返回（不管发没发出去）
  │  recv()  ──→  从内核接收缓冲区取数据  ──→  返回（不管到没到齐）
  │
  ▼
内核层
  └── TCP 栈：什么时候发、分几个包、重不重传——全是内核说了算
```

- **send()** 只承诺"数据拷进内核发送缓冲区了"，不承诺"数据发出去了"。什么时候真正发、分几个包发、重不重传，全是内核 TCP 栈说了算——应用层完全不知情。这就是异步：你把活交给内核，它什么时候干完不告诉你。
- **recv()** 同理：数据什么时候到、分几次到、一次 recv 能拿多少，全不归你管。

### 粘包与拆包

更麻烦的是，TCP 是**字节流**，不保留消息边界：

```
发送端： write("hello")  × 3

接收端可能拿到：
  ┌─────────────────────┐
  │ "hellohellohello"    │  ← 粘包（三次合并成一次）
  │ "hel" + "lohello..." │  ← 拆包（一次拆成多次）
  └─────────────────────┘
```

**你需要一个应用协议来界定消息边界。**

## 3. 应用层消息界定格式

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **分隔符界定** | 特殊字节标记消息结尾（如 `\r\n`） | 简单，可读性好 | 不适用于二进制（数据中可能出现分隔符） |
| **前置长度** | 固定字节 header 声明消息体长度 | 简单，适合二进制 | 需要提前知道长度 |
| **固定长度** | 所有消息字节数完全相同 | 实现最简单 | 浪费空间，灵活性差 |

## 4. RESP 格式

Redis 选择了**混合策略**：首字符界定类型 + 分隔符 + 前置长度按需组合。

| 首字节 | 类型 | 格式 | 示例 |
|--------|------|------|------|
| `+` | Simple String | `+<文本>\r\n` | `+OK\r\n` |
| `-` | Error | `-<错误信息>\r\n` | `-ERR unknown command\r\n` |
| `:` | Integer | `:<数字>\r\n` | `:1000\r\n` |
| `$` | Bulk String | `$<长度>\r\n<数据>\r\n` | `$5\r\nhello\r\n` |
| `*` | Array | `*<元素数>\r\n<元素>...` | `*2\r\n$3\r\nfoo\r\n$3\r\nbar\r\n` |

**设计思路：**

- **首字符**用来快速区分类型，一个字节完成分派
- **短文本场景**（Simple String、Error、Integer）：`\r\n` 分隔符足够简洁，一次界定边界的消耗几乎忽略不计，也不会有二进制问题
- **二进制/长串场景**（Bulk String）：使用**前置长度**，简单可靠，适合任意二进制数据

这就是 RESP 的精妙之处——不是死板地选一种，而是**按类型选用最合适的策略**。

## 5. 为什么 epoll 适合高并发？和 select/poll 的区别是什么？

### 先看 select/poll 有什么缺陷

```c
while (1) {
    FD_ZERO(&rfds);
    FD_SET(fd1, &rfds);   // 每次循环都要重新注册所有 fd
    FD_SET(fd2, &rfds);
    // ...

    select(maxfd + 1, &rfds, NULL, NULL, NULL);

    for (int i = 0; i <= maxfd; i++) {   // 遍历所有 fd
        if (FD_ISSET(i, &rfds)) {
            // 处理
        }
    }
}
```

**两个致命问题：**

1. **每次都要重新传递 fd 集合** — 内核与应用层之间大量数据拷贝
2. **O(n) 遍历** — select 返回后必须遍历所有 fd 才能找到就绪的那几个，大部分遍历是无效的

### epoll 如何解决：事件驱动

epoll 不会让你去主动遍历寻找，而是**直接告诉你哪些 fd 触发了事件**。

```
epoll fd 内部结构：

  ┌── 红黑树 (rbtree) ───── 存储所有注册的 fd，key = fd
  │                          （用于 epoll_ctl 的增删改查）
  │
  └── 就绪链表 (rdllist) ─── 存储就绪的 fd
                              （epoll_wait 直接从这里取，O(1)）
```

**工作流程：**

```
网卡中断 → 内核将数据放入缓冲区
         → 调用 ep_poll_callback() 回调
         → 将 fd 丢进 rdllist（就绪链表）
         → epoll_wait 返回，直接取 rdllist 中的 fd
```

| | select | poll | epoll |
|------|--------|------|-------|
| 数据结构 | 位图 (fd_set) | 链表 | 红黑树 + 就绪链表 |
| fd 上限 | 1024（可改但重编译） | 无限制 | 无限制 |
| 每次注册 fd | 是（重新传整个集合） | 是 | 否（增量增删，epoll_ctl） |
| 获取就绪 fd | O(n) 遍历 | O(n) 遍历 | O(1) 直接从就绪链表取 |
| 内核-用户态拷贝 | 大量 | 大量 | 只拷贝就绪的 |
| 设计思想 | 轮询 | 轮询 | **事件驱动（观察者模式）** |

> epoll 的设计思想和设计模式中的**观察者模式**本质相同——"别问我，我通知你"。

## 6. 水平触发 vs 边沿触发：Redis 为什么选择水平触发（LT）？

| | 水平触发 (LT) | 边沿触发 (ET) |
|------|-------------|-------------|
| 触发条件 | 缓冲区有数据就通知 | 缓冲区状态变化时才通知一次 |
| 半包处理 | 数据没读完下次继续通知 | 数据没读完**不会再通知**，直到新数据到达 |
| 实现复杂度 | 简单 | 需要循环读直到 EAGAIN |

### Redis 选 LT 的原因

**边沿触发遇到半包会导致数据饥饿**：如果一次没读完，fd 不再被触发，剩余数据就"卡"在里面了——直到新数据到来才会被重新触发。

对于 Redis 这种**单线程模型**来说，任何阻塞或饥饿都是致命的。Redis 在其他地方的设计也能看出它在主动规避阻塞：

| 场景 | 规避策略 |
|------|---------|
| 网络 I/O | 事件驱动 + 水平触发 |
| 持久化 | `fork()` + COW 写时复制（子进程干活，主进程不阻塞） |
| 数据读写 | 单线程，避免锁争夺开销 |
| 日志写入 | 重定向符异步写入 |
| 惰性删除 | 随机抽样，在主循环中顺手做，不交给中断 |

> 水平触发的好处：**实现简单 + 避免半包阻塞**。在 Redis 单线程模型下，LT 的额外唤醒开销远小于 ET 饥饿带来的风险。
