今天我们只讲一个问题：**单线程为什么快，如何设计高速的单线程系统。**

---

## 1. 先打一个补丁

不是"单线程 = 快"，而是在特定场景下，单线程可以快到极致。

快的本质是 **CPU 利用率**。CPU 是心脏，利用率越高，吞吐越高。

---

## 2. 为什么业务系统多线程，Redis 却单线程？

这很反直觉，答案在阻塞模型的不同。

- **业务系统**：操作数据库，IO 阻塞频繁。一个线程卡住，其他线程顶上——多线程把 CPU 填满。
- **内存 KV**：存取几乎无阻塞。引入多线程反而带来锁竞争、缓存行失效，得不偿失。

与其困在锁上，不如拿了数据直接走。

---

## 3. 高速单线程的设计核心：委托

委托，就是把阻塞的活交给别人——操作系统、硬件、子进程，甚至是未来的自己。

Redis 的快，正是因为把几乎所有阻塞都委派了出去：

| 机制 | 委托对象 | 委派了什么 |
|------|----------|------------|
| **epoll** | 内核 + 网卡 | 网络消息的等待与分发 |
| **RDB / AOF** | 子进程 | 磁盘 IO 的持久化写入 |
| **LOG 日志** | 操作系统 | 文件缓冲区的实际落盘 |
| **TTL 过期** | 空间 + 未来 + 抽样 | 过期 key 的清理检查 |
| **ReHash** | 未来的自己 | 渐进式搬桶，分摊到每次请求 |

---

## 4. 总结

纵观 Redis 架构，设计哲学一以贯之：

> **把阻塞交给别人，把 CPU 留给自己。**

所以别再背"Redis 快是因为单线程"这种八股了。

Redis 快的真正原因是——**CPU 利用率高**。
